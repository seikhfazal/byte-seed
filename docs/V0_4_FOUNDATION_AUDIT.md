# ByteSeed v0.4 Foundation Audit

Audit date: 2026-07-15
Branch: `v0.4-foundation-audit`
Audited commit: `6fc1cf5` (`v0.3-speed-polish`; also the current `main` tip)
Stable local checkpoint: `checkpoints/anchor_v2_3_finetuned.pt`

## 1. Executive summary

ByteSeed has a clear, compact educational GPT implementation, sound basic tensor dimensions, a correct lower-triangular causal mask, tied token/output embeddings, example-wise SFT, functional CUDA fp32/fp16 inference, and sensible artifact exclusions. The current checkpoint loads and retains all nine deliberately reinforced anchor behaviors.

The foundation is not yet ready to support claims of held-out model quality or exact training reproducibility. The highest-impact confirmed problems are: an SFT truncation path that can produce an all-ignored `NaN` loss; an off-by-one token-batch sampler; pretraining resume choosing an unrelated SFT checkpoint with no optimizer; incomplete exact-resume state; no tokenizer/corpus fingerprints; complete verbatim overlap between stable regression prompts and SFT data; malformed and stale tracked SFT examples; and nondeterministic, text-only evaluation/benchmark reporting.

Required interpretation of model quality:

- **Anchor-retention regression: 9/9.**
- **Held-out generalization: not yet measured.**

All nine stable prompts occur verbatim in the Anchor v2.3 SFT data. The 9/9 result is evidence of anchor retention only, not broad generalization.

The lowest-risk, highest-value next implementation is **PR 1: deterministic unit-test foundations and CPU GitHub Actions**. No recommendation was implemented during this audit.

## 2. Scope and restrictions

The audit inspected every tracked source file, script, configuration, document, example, raw dataset, package manifest, ignore rule, and release document. Ignored artifacts were not changed. Only trusted locally generated ByteSeed checkpoint metadata was loaded, with `map_location="cpu"` for metadata inspection. Tokenizer binaries were read only for vocabulary and special-token checks.

No production code, model architecture, dataset, tokenizer artifact, checkpoint, or training state was modified. No builder, tokenizer training, data preparation, pretraining, SFT, optimizer-step smoke test, download, push, merge, tag, or release command was run. The only repository writes are the four requested skills and this audit document.

## 4. Current strengths

- **Coherent model dimensions.** `configs/byteseed_12m.yaml:1-6` defines 8 layers, 8 heads, width 320, and block size 256; each head is 40 dimensions. `src/byteseed/config.py:80-96` validates positive heads, head divisibility, split bounds, block size, vocabulary, batch size, and accumulation.
- **Straightforward educational architecture.** `src/byteseed/model.py:12-37`, `src/byteseed/model.py:40-66`, and `src/byteseed/model.py:69-103` clearly implement QKV attention, pre-norm residual blocks, a 4x GELU MLP, final LayerNorm, logits, and cross-entropy.
- **Correct causal-mask construction by inspection.** `src/byteseed/model.py:21-35` registers a lower-triangular mask, slices it to the active sequence, fills future scores with `-inf`, and normalizes along the key dimension.
- **Weight tying is explicit.** `src/byteseed/model.py:73-80` aliases `lm_head.weight` to `token_embedding.weight`, contributing to the observed 11,129,920 parameters.
- **Sequence bounds are explicit.** `src/byteseed/model.py:90-99` rejects input longer than the learned positional table and constructs positions on the input device.
- **Single-example SFT boundaries are much safer than random chunking.** `src/byteseed/finetune_chat.py:23-64` keeps examples independent, shifts labels once, and masks prompt labels; `src/byteseed/finetune_chat.py:66-81` masks padding.
- **Training includes useful stability controls.** `src/byteseed/pretrain.py:72-87` implements configured gradient accumulation, CUDA autocast/scaling, and gradient clipping. SFT also clips and scales at `src/byteseed/finetune_chat.py:94-108`.
- **Tokenizer alignment and diagnostics exist.** `src/byteseed/config.py:131-140` aligns new models to the effective vocabulary; `src/byteseed/generate.py:27-70` reports vocabulary/embedding mismatch context. Current special tokens round-trip, with IDs 4–7 appearing in encoded sequences.
- **Inference paths are practical.** `src/byteseed/model.py:105-141` uses inference mode and context cropping; `src/byteseed/chat.py:61-76` selects fp16 only on CUDA; `src/byteseed/chat.py:137-159` formats prompts and removes stop/legacy label artifacts; history remains opt-in and capped at two turns at `src/byteseed/chat.py:307-318` and `src/byteseed/chat.py:347-382`.
- **Benchmark timing handles CUDA correctly.** `scripts/benchmark_generation.py:23-25` synchronizes CUDA, `scripts/benchmark_generation.py:104-110` performs explicit warm-ups and resets peak memory, and `scripts/benchmark_generation.py:114-146` synchronizes timed runs and reports latency, tokens/s, and peak memory.
- **Focused Anchor v2.3 data has local validation.** `scripts/build_anchor_v2_3_sft.py:150-163` validates rows and categories; `scripts/build_anchor_v2_3_sft.py:167-207` rejects exact duplicates and selected forbidden phrases. The inspected v2.3 file has 80 rows and no exact user/answer duplicates.
- **Honest project positioning and artifact hygiene.** `README.md:37-42`, `README.md:162-170`, and `docs/LIMITATIONS.md:3-11` describe a small educational model; `.gitignore:1-37` excludes environments, checkpoints, tokenizer binaries, processed/generated data, and runs. Required `git check-ignore` checks resolved to the intended rules.

## 5. Confirmed defects

### CD-01 — SFT right truncation can remove every supervised answer label

- **Classification:** Confirmed defect
- **Location:** `src/byteseed/finetune_chat.py:44-64`; `src/byteseed/model.py:101-102`
- **Concrete evidence:** `_encode_example` concatenates prompt and answer, keeps the leftmost `block_size + 1` tokens, then masks up to `len(prompt_ids) - 1`. A read-only targeted inspection with the current tokenizer and a long prompt returned 256 input tokens and **0 supervised labels**. PyTorch cross-entropy with every target equal to `-100` returned `NaN`. The current 80 Anchor v2.3 examples are not affected: they retain 17–41 supervised labels.
- **Likely user-visible impact:** A long SFT prompt can silently inject `NaN` loss and destabilize or invalidate a fine-tuning run.
- **Smallest safe corrective change:** Reserve space for at least one answer token plus `<|end|>`; truncate the prompt side first, and reject any encoded example with zero non-ignored targets before training.
- **Validation required:** Boundary tests for prompt lengths below, at, and above the block limit; assert at least one supervised target; assert finite loss/backward; verify current v2.3 encodings remain unchanged where they already fit.
- **Backward compatibility:** Checkpoint shape and normal examples remain unchanged. Only previously invalid overlong examples change or are rejected.

### CD-02 — TokenDataset rejects a length it accepts and omits the last valid window

- **Classification:** Confirmed defect
- **Location:** `src/byteseed/dataset.py:20-32`
- **Concrete evidence:** Construction accepts `len(data) == block_size + 1`, but `torch.randint(len(data) - block_size - 1, ...)` then receives an exclusive upper bound of zero. A read-only inspection with 33 tokens and block size 32 reproduced `RuntimeError: random_ expects 'from' to be less than 'to'`. For longer arrays the same bound excludes the final valid start index.
- **Likely user-visible impact:** Minimal valid splits crash; all other splits never sample their last valid next-token window.
- **Smallest safe corrective change:** Use the number of valid starts, `len(data) - block_size`, as the exclusive upper bound and keep the constructor requirement `len(data) > block_size`.
- **Validation required:** Tests at lengths `block_size`, `block_size + 1`, and `block_size + 2`; enumerate valid starts with a fixed seed; verify `x/y` shifting and shapes.
- **Backward compatibility:** Shapes and ordinary batches are unchanged; sampling gains the previously omitted final window.

### CD-03 — Pretraining resume selects the newest unrelated SFT checkpoint

- **Classification:** Confirmed defect
- **Location:** `src/byteseed/utils.py:24-26`; `src/byteseed/pretrain.py:46-69`; `src/byteseed/finetune_chat.py:109-111`
- **Concrete evidence:** `latest_checkpoint` sorts every `*.pt` only by modification time. On this repository it returns `checkpoints/anchor_v2_3_finetuned.pt`. Metadata inspection shows that checkpoint contains only `config`, `iter`, and `model`; line 69 of pretraining unconditionally reads `ckpt["optimizer"]`.
- **Likely user-visible impact:** `python -m src.byteseed.pretrain --resume` fails before training with a missing-optimizer error whenever a newer SFT checkpoint is present.
- **Smallest safe corrective change:** Add a versioned checkpoint `kind`; require `kind == "pretrain"` plus optimizer state for resume; accept an explicit `--resume-checkpoint`; filter legacy candidates by required keys rather than modification time.
- **Validation required:** Temporary local fixtures for pretrain, SFT, malformed, and legacy schemas; prove selection ignores newer SFT files and emits actionable errors.
- **Backward compatibility:** Existing pretraining checkpoints load through a documented legacy-key fallback. Existing SFT checkpoints remain inference-compatible but are rejected for pretraining resume.

### CD-04 — Resume cannot reproduce an uninterrupted run exactly

- **Classification:** Confirmed defect
- **Location:** `src/byteseed/utils.py:10-15`; `src/byteseed/pretrain.py:38-71`; `src/byteseed/pretrain.py:89-101`
- **Concrete evidence:** Training reseeds at startup, then checkpoints save only `model`, `optimizer`, `config`, `iter`, and `best_val`. They omit Python, NumPy, Torch CPU/CUDA RNG, AMP scaler, and `patience_left`. The learning-rate policy is a stateless function of iteration, so there is no separate scheduler object to save. Regular iteration checkpoints save `best_val` on line 93 **before** updating it on lines 94–96.
- **Likely user-visible impact:** A resumed trajectory samples different batches/dropout masks, may use a reset scaler/patience budget, and may carry stale best-loss metadata. Results cannot be reproduced exactly from a checkpoint.
- **Smallest safe corrective change:** Save/restore all RNG states, scaler state, current patience, iteration, updated best validation loss, and a scheduler-policy/version field; update `best_val` before writing the regular checkpoint.
- **Validation required:** CPU interrupted-versus-uninterrupted parity through several steps; CUDA parity where available; best-loss and patience transition tests; legacy fallback tests.
- **Backward compatibility:** Load old checkpoints with warnings and initialized defaults; write a new schema version without changing model parameter keys.

### CD-05 — Checkpoint/tokenizer compatibility is based on shape, not token identity

- **Classification:** Confirmed defect
- **Location:** `src/byteseed/train_tokenizer.py:43-54`; `tokenizer/tokenizer_meta.json:1-13`; `src/byteseed/pretrain.py:89-97`; `src/byteseed/finetune_chat.py:109-111`; `src/byteseed/generate.py:27-85`
- **Concrete evidence:** Tokenizer metadata records requested/target/actual sizes, symbols, and paths but no tokenizer hash, corpus hash, SentencePiece version, or construction manifest. Checkpoints store no tokenizer/corpus identifier. Loading checks embedding dimensions only; a different 3,699-piece tokenizer could pass shape checks while mapping IDs to different text.
- **Likely user-visible impact:** A same-sized but semantically different tokenizer can load successfully and produce meaningless output; a public user cannot verify which tokenizer/corpus belongs to a checkpoint.
- **Smallest safe corrective change:** Compute stable SHA-256 fingerprints for tokenizer model, ordered vocabulary, corpus/manifest, and special-token IDs; save them in tokenizer metadata and checkpoint provenance; reject mismatches for new schemas and warn for legacy checkpoints.
- **Validation required:** Exact-match load, same-size/different-tokenizer rejection, corpus-change detection, special-token ID checks, and legacy warning tests.
- **Backward compatibility:** Existing checkpoints stay loadable under an explicit legacy policy; new checkpoints gain strict identity checks.

### CD-06 — Stable regression is fully contaminated by verbatim SFT prompts

- **Classification:** Confirmed defect
- **Location:** `scripts/eval_stable_v0_2.py:22-31`; `data/raw/assistant_sft/anchor_v2_3_patch.jsonl:1-1`; `data/raw/assistant_sft/anchor_v2_3_patch.jsonl:19-19`; `data/raw/assistant_sft/anchor_v2_3_patch.jsonl:36-36`; `data/raw/assistant_sft/anchor_v2_3_patch.jsonl:71-76`
- **Concrete evidence:** All nine evaluation prompts appear verbatim as SFT user fields; the same rows are released at `examples/byteseed_anchor_v2_3_sft.jsonl:1-1`, `examples/byteseed_anchor_v2_3_sft.jsonl:19-19`, `examples/byteseed_anchor_v2_3_sft.jsonl:36-36`, and `examples/byteseed_anchor_v2_3_sft.jsonl:71-76`.
- **Likely user-visible impact:** A 9/9 score can be mistaken for broad ability even though it measures retention of trained anchors.
- **Smallest safe corrective change:** Rename/report this suite as anchor retention, add automated exact/normalized overlap warnings, and create a separately versioned held-out paraphrase suite excluded by dataset manifests.
- **Validation required:** Assert the current suite reports 9/9 overlap; deliberate overlap must warn/fail according to mode; the held-out suite must report zero exact/normalized training matches.
- **Backward compatibility:** Preserve the nine prompts and existing pass rules for retention history; change labels and add a separate generalization metric.

### CD-07 — Regression failures do not produce a failing process status

- **Classification:** Confirmed defect
- **Location:** `scripts/eval_stable_v0_2.py:127-138`
- **Concrete evidence:** The script prints pass/fail results and a summary but never exits nonzero when `passed < len(PROMPTS)`.
- **Likely user-visible impact:** CI or automation can report success even when model regressions fail.
- **Smallest safe corrective change:** Return a result object from evaluation and make the CLI exit nonzero on failed anchors, with an optional report-only mode if needed.
- **Validation required:** Stubbed pass and failure cases asserting process status and machine-readable counts.

### CD-08 — Batched generation cannot stop on stop tokens

- **Classification:** Confirmed defect
- **Location:** `src/byteseed/model.py:117-140`
- **Concrete evidence:** Stop handling is guarded by `next_id.numel() == 1`. For every batch larger than one, `numel()` is greater than one, so the stop condition can never run.
- **Likely user-visible impact:** Batched callers continue generating after `<|end|>`/`<|user|>`, wasting work and returning unwanted continuation tokens.
- **Smallest safe corrective change:** Track a finished mask per sequence, force a defined pad/stop token for finished rows, and exit once all rows finish. If that policy is deferred, reject batched generation with stop IDs explicitly.
- **Validation required:** Mixed-stop-time batches, all-stop batches, no-stop batches, batch-one parity, and output-padding policy tests.
- **Backward compatibility:** Batch-one behavior remains unchanged; batched output termination becomes defined.

## 6. Suspected risks

### SR-01 — Manual attention fp16 numerical behavior lacks parity evidence

- **Classification:** Suspected risk
- **Location:** `src/byteseed/model.py:25-37`; `src/byteseed/pretrain.py:77-87`; `src/byteseed/finetune_chat.py:97-108`
- **Concrete evidence:** Scores and softmax are manually composed under CUDA autocast. Short fp16 inference succeeds, but no safe existing test compares fp16 logits/attention outputs/gradients to fp32 or exercises extreme score magnitudes.
- **Likely user-visible impact:** Longer or extreme inputs could produce precision drift or non-finite values during training/inference.
- **Smallest safe corrective change:** Add finite-value and tolerance tests first; keep manual attention as default. Consider optional SDPA only after parity is established.
- **Validation required:** fp32/fp16 forward and backward comparisons, long-context causal tests, finite loss/gradient assertions, and representative CUDA hardware coverage.
- **Backward compatibility:** Tests do not change behavior; any SDPA path must remain optional with manual default.

### SR-02 — Sliding context and future KV caching have unresolved positional semantics

- **Classification:** Suspected risk
- **Location:** `src/byteseed/model.py:90-99`; `src/byteseed/model.py:118-120`
- **Concrete evidence:** Generation crops to the last block and recomputes positions from zero. This is internally consistent for the current uncached path, but a cache cannot simply append indefinitely because learned positions span only 0–255 and a rolling window changes every retained token's assigned position.
- **Likely user-visible impact:** A naive KV cache could diverge from current outputs or use invalid positions after the context limit.
- **Smallest safe corrective change:** Add `use_kv_cache=False` by default; track per-layer K/V and absolute-in-window positions; on window rollover, rebuild the cache from the retained window before optimizing further.
- **Validation required:** Per-step cached/uncached logit parity, fixed-seed token parity, rollover at 255/256/257 tokens, reset/batch tests, and memory/latency reports.
- **Backward compatibility:** Keep current uncached generation and checkpoint parameter keys unchanged.

### SR-03 — Contiguous token splitting may leak document fragments across train/validation

- **Classification:** Suspected risk
- **Location:** `src/byteseed/dataset.py:9-17`; `src/byteseed/prepare_data.py:13-25`
- **Concrete evidence:** All top-level Markdown is concatenated into one string, tokenized once, and sliced at a 90% token index. No document IDs, group split, boundary manifest, or duplicate-document check is used. This demonstrates a non-boundary-aware method, but the exact current boundary was not regenerated or modified during this audit.
- **Likely user-visible impact:** A document or repeated template family may straddle the split, making validation loss optimistic and semantically contaminated.
- **Smallest safe corrective change:** Split source documents/groups deterministically before token concatenation; store ordered source lists and hashes; keep complete documents or declared chunks in one split.
- **Validation required:** Disjoint source/group assertions, boundary reports, exact/near-duplicate checks across splits, deterministic manifests, and token-count comparisons.
- **Backward compatibility:** Existing processed arrays/checkpoints remain historical artifacts; new split manifests define a new data version.

### SR-04 — Glob-based SFT assembly can mix historical patches, tests, and generated categories

- **Classification:** Suspected risk
- **Location:** `scripts/build_personal_dataset.py:41-64`; `scripts/build_personal_dataset.py:67-92`
- **Concrete evidence:** The builder includes every `data/raw/assistant_sft/*.jsonl` and every generated SFT file by default. That selection includes identity smoke data, historical anchor patches, malformed v2.1 rows, and category-heavy generated data. No versioned inclusion manifest is used.
- **Likely user-visible impact:** Future combined datasets may contain unintended duplication, contradictions, evaluation prompts, and category imbalance.
- **Smallest safe corrective change:** Replace directory-wide globs with an explicit versioned manifest containing paths, roles, licenses, hashes, weights, and excluded evaluation files.
- **Validation required:** Manifest schema/hash checks, deterministic row count, category distribution, duplicate/contradiction report, and exclusion of held-out data.
- **Backward compatibility:** Preserve the old combined file as a historical data version; do not silently reuse its name for new composition.

### SR-05 — Near-duplicate, contradiction, and category-leakage detection is too narrow

- **Classification:** Suspected risk
- **Location:** `scripts/inspect_dataset.py:33-35`; `scripts/inspect_dataset.py:50-79`; `scripts/inspect_dataset.py:159-188`; `scripts/inspect_sft_file.py:25-45`
- **Concrete evidence:** Inspection counts exact strings and fixed prefixes. It has no normalized-punctuation match, token n-gram similarity, prompt-family grouping, contradictory-answer rule, or cross-category leakage matrix. The authorized run found 210 exact duplicate-user excess and 95 exact duplicate-answer excess, so broader similarity is plausible but not measured.
- **Likely user-visible impact:** Template dominance and semantic duplicates may survive inspection and inflate dataset/evaluation coverage.
- **Smallest safe corrective change:** Add deterministic normalized exact matching plus lightweight token-shingle similarity and category/prompt-family reports; require manual review for semantic contradiction flags.
- **Validation required:** Synthetic exact/normalized/near-duplicate fixtures, false-positive fixtures, category leakage examples, and stable report ordering.
- **Backward compatibility:** Reporting only; thresholds must be versioned before becoming hard gates.

### SR-06 — Legacy checkpoint compatibility is inferred incompletely

- **Classification:** Suspected risk
- **Location:** `src/byteseed/generate.py:27-56`; `src/byteseed/generate.py:59-85`; `src/byteseed/config.py:124-128`
- **Concrete evidence:** For a checkpoint without config, only vocabulary and embedding width are inferred from one tensor; layer count, head count, MLP layout, block size, schema kind, and tokenizer identity come from the YAML/defaults. Current known checkpoints contain config, so a failing legacy case was not loaded during this audit.
- **Likely user-visible impact:** Some legacy checkpoints may fail with generic shape errors or, for same-shaped tokenizers, load with wrong semantics.
- **Smallest safe corrective change:** Version schemas; validate all architecture fields/tensor keys; provide explicit legacy adapters for known formats and fail closed with actionable messages otherwise.
- **Validation required:** Fixture matrix for current, missing-config, missing-key, changed-depth/head/block, and tokenizer-mismatch checkpoints.
- **Backward compatibility:** Known legacy formats remain supported through adapters; unknown formats get clearer rejection.

### SR-07 — SFT ignores configured gradient accumulation

- **Classification:** Suspected risk
- **Location:** `src/byteseed/config.py:44-46`; `src/byteseed/finetune_chat.py:94-108`
- **Concrete evidence:** The shared config exposes `gradient_accumulation_steps`, and pretraining honors it, but SFT performs one batch and one optimizer step per iteration. Documentation does not state whether this is intentional.
- **Likely user-visible impact:** SFT effective batch size and memory behavior differ from what users may infer from the shared config.
- **Smallest safe corrective change:** Either document SFT's independent semantics or implement accumulation behind an explicit SFT option while preserving current default behavior.
- **Validation required:** Effective-batch equivalence test, loss-scaling/clip ordering test, and documented iteration/optimizer-step counters.
- **Backward compatibility:** Keep existing SFT default unless a versioned training-policy change is intentional.

### SR-08 — Public environment reproduction is weak because dependencies are unpinned

- **Classification:** Suspected risk
- **Location:** `requirements.txt:1-5`; `requirements-data.txt:1-1`; `pyproject.toml:1-25`

## 10. Performance opportunities

### Optional SDPA design

Keep `CausalSelfAttention` in `src/byteseed/model.py` as the default educational implementation. Add an explicitly selected runtime attention implementation whose SDPA branch reshapes the existing Q/K/V tensors identically and calls `torch.nn.functional.scaled_dot_product_attention` with causal behavior and training-only dropout. Do not change parameter names or checkpoint tensors. If the selector enters saved config, loaders must tolerate the field in both directions or keep it in runtime metadata instead.

Acceptance requires fp32 numerical tolerance, fp16 finite behavior, causal-prefix invariance, dropout-mode checks, identical output shapes, forward/backward checks, manual-default confirmation, and checkpoint load parity. Report performance only after deterministic warm-ups and enough runs.

### Optional KV-cache design

Add `use_kv_cache=False` as the compatibility default. Cache per-layer keys/values and return updated caches without changing learned parameters. Use the correct position for each newly appended token. Because current rolling-window generation reindexes the full retained window from zero, rebuild caches when crossing the context window until an explicitly tested alternative positional policy exists.

Acceptance requires per-step cached/uncached logit parity, fixed-seed token parity, mixed stop times, batch-one and batch-many behavior, cache reset, 255/256/257-token boundaries, memory/latency reporting, and unchanged uncached output/checkpoint behavior.
