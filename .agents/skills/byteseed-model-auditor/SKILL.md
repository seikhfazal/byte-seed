# ByteSeed Model Auditor

Perform a conservative, evidence-based audit without changing production behavior.

## Safety boundary

- Default to read-only inspection. Do not edit production code, model architecture, datasets, tokenizer artifacts, or checkpoints.
- Do not train, pretrain, run SFT, prepare data, run optimizer-step smoke tests, download artifacts, push, merge, tag, or release.
- Load only trusted locally generated ByteSeed checkpoints and prefer `map_location="cpu"` for metadata.
- Use only existing safe checks authorized by the task. Do not use `torch.compile` unless explicitly requested.

## Audit workflow

1. Confirm branch, Git status, package metadata, configurations, docs, and all tracked files.
2. Trace tensor shapes through token/position embeddings, attention heads, projections, residuals, LayerNorm, MLP, final normalization, tied LM head, logits, and shifted loss.
3. Verify manual causal masking, sequence limits, context truncation, numerical behavior, and future-token isolation from code or safe tests.
4. Trace generation end to end: prompt encoding, batching, temperature, top-k, repetition penalty, stop tokens, retry cleanup, history/stateless prompts, and context limits.
5. Trace checkpoint save/load/resume: selection, type, schema, config, tensor shapes, tokenizer identity, optimizer/scaler/RNG/scheduler/patience state, iteration, best validation loss, provenance, and legacy handling.
6. Trace pretraining and SFT loops: batching, learning rate, accumulation, clipping, AMP, validation, early stopping, label shifting, ignored labels, prompt masking, truncation, padding, and all-ignored targets.
7. Compare fp32/fp16 paths and benchmark controls. Record seeds, sampling, warm-up, synchronization, token counting, memory, and output format.
8. Inventory conventional tests and CI. Map every uncovered invariant.
9. Describe SDPA and KV-cache opportunities only as optional, compatibility-preserving designs. Keep manual attention and uncached generation as defaults; require numerical, causal, shape, positional, context-limit, and cached/uncached parity tests.

## Evidence rules

Classify each item as `Strength`, `Confirmed defect`, `Suspected risk`, or `Missing test`. Confirm a defect only when code, repository data, safe output, or an existing non-writing validation demonstrates the broken invariant. Otherwise identify the concrete code path and targeted test needed.

For every defect, risk, and missing test provide:

- `path:start-end` using current-branch line numbers;
- concrete evidence;
- likely user-visible impact;
- smallest safe corrective change;
- validation required;
- backward-compatibility implications.

Never describe anchor-retention results as generalization. Report `Anchor-retention regression` separately from `Held-out generalization`, and explicitly disclose train/eval overlap.

## Expected output

Produce an audit with scope, baseline, strengths, defects, risks, missing tests, data/evaluation findings, checkpoint/reproducibility findings, performance opportunities, public-release limitations, ranked improvements with effort/risk/exact files, validation evidence, and a sequence of small pull requests. Do not implement recommendations unless separately authorized.
